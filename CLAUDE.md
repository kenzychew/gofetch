# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project Overview

GoFetch is a production-grade RAG (Retrieval-Augmented Generation) system featuring hybrid search (BM25 + dense vectors), cross-encoder re-ranking, streaming generation with SSE, and inline citation grounding. Deployed via Docker Compose.

## Tech Stack

- **Backend**: FastAPI with SSE streaming (uvicorn)
- **Frontend**: Gradio
- **Vector DB**: PostgreSQL with pgvector (Docker container)
- **Sparse Search**: rank_bm25 (in-memory, pickled to disk)
- **Embeddings**: sentence-transformers (`all-MiniLM-L6-v2`, local)
- **Re-ranker**: cross-encoder (`ms-marco-MiniLM-L-6-v2`, local)
- **LLM**: Anthropic API (Claude Haiku) via streaming
- **Chunking**: LangChain text splitters (recursive character, chunk_size=512, overlap=50)
- **Python**: 3.11+, dependency management via `uv`

## Architecture

The system is a three-stage pipeline:

1. **Ingestion** (`src/ingestion/`): Load documents (PDF/text) -> chunk -> embed with sentence-transformers -> upsert into PostgreSQL (pgvector) + build BM25 index
2. **Retrieval** (`src/retrieval/`): Query -> parallel BM25 + dense vector search (top 20 each) -> RRF fusion (top 10) -> cross-encoder re-rank (top 5)
3. **Generation** (`src/generation/`): Build citation-aware prompt from top chunks -> stream Anthropic response with `[1]`, `[2]` inline citations

**API endpoints** (`src/api/main.py`):
- `POST /ingest` — upload and index documents
- `GET /query` — SSE streaming answer with citations
- `GET /health` — health check

**Key design decisions**:
- Hybrid search (BM25 + dense) with Reciprocal Rank Fusion (k=60) catches both keyword-exact and semantic matches
- Two-stage retrieval: fast bi-encoder first stage, accurate cross-encoder second stage on small candidate set
- BM25 index is in-memory via rank_bm25 (pickled to disk), not a separate service
- All embeddings and re-ranking run locally (no API calls), only generation uses Anthropic API

- Imports: Always use top-level imports. Never use lazy/deferred imports inside functions or methods.

  - Avoid wildcard imports (eg `from module import *`).

- Text: Always use plain keyboard characters in code, comments, docstrings, and documentation. Use `->` not arrow symbols, `eg` or `eg.` not `e.g.`, `--` not em-dashes, straight quotes `"` not curly quotes, etc. No Unicode arrows, dashes, or typographic symbols.
- Git commits: Never include Claude/AI references in commit messages (no Co-Authored-By, no "Generated by", etc.)

### Pydantic vs Dataclass vs Class

**Use Pydantic for:**
- API request/response schemas (FastAPI integration)
- Configuration management
- Data validation and parsing
- Any data structure requiring validation

**Use Dataclass for:** Simple data containers without validation

**Use Regular class for:** Stateful objects with multiple methods operating on data

**Don't use class when:** Just storing data (use dataclass), just grouping functions (use module), single method called once (use function)

```python
# Pydantic: For validation and API schemas
class UserInput(BaseModel):
    email: str
    age: int = Field(ge=0, le=150)

# Dataclass: For simple data storage
@dataclass
class UserData:
    name: str
    email: str
    age: int

# Regular class: For stateful objects with methods
class UserProcessor:
    def __init__(self, config: Config):
        self.config = config
        self.cache = {}

    def process(self, user: UserData) -> ProcessedUser:
        ...
```

```python
# Bad: Unnecessary class -- single method called once
class DataProcessor:
    def __init__(self, config):
        self.config = config

    def process(self, data):
        return data * self.config

processor = DataProcessor(config)
result = processor.process(data)

# Good: Just use a function
def process_data(data, config):
    return data * config

result = process_data(data, config)

# Good: Class with state and multiple methods
class CacheManager:
    def __init__(self, max_size: int):
        self.max_size = max_size
        self.cache: dict = {}
        self.access_count: dict = {}

    def get(self, key: str) -> str | None:
        if key in self.cache:
            self.access_count[key] += 1
            return self.cache[key]
        return None

    def put(self, key: str, value: str) -> None:
        if len(self.cache) >= self.max_size:
            self._evict_least_used()
        self.cache[key] = value
        self.access_count[key] = 1
```

### Inheritance and Subclassing

- **Prefer `super()`**: When subclassing, prioritize using the base class's method via `super().method_name()` unless an override is absolutely required
- **Abstract base classes**: Must inherit from `abc.ABC` and use the `@abstractmethod` decorator
- **Avoid unnecessary overrides**: Do not redefine a method that the parent already implements correctly -- extend behavior with `super()` instead

```python
# Bad: Overriding base class method entirely
class ChildProcessor(BaseProcessor):
    def process(self, data):
        # Duplicates logic from BaseProcessor.process
        validated = self.validate(data)
        result = self.transform(validated)
        return result

# Good: Extend via super()
class ChildProcessor(BaseProcessor):
    def process(self, data):
        result = super().process(data)
        # Only add child-specific behavior
        return self.post_process(result)
```

```python
# Good: Proper abstract base class
from abc import ABC, abstractmethod

class BaseProcessor(ABC):
    @abstractmethod
    def transform(self, data: dict) -> dict:
        ...

    def process(self, data: dict) -> dict:
        validated = self.validate(data)
        return self.transform(validated)
```

### Complexity Thresholds
- **Functions**: Target 15-20 lines (max 50), limit 5 parameters, max 12 branches
- **Classes**: Single clear responsibility, methods should use `self` (otherwise extract to functions)
- **Modules**: Keep files focused on single domain, avoid circular imports, use explicit imports (no wildcards)
- **Logic**: Simplify complex booleans with intermediate variables
- **Use early returns** to reduce nesting -- no deep if/else chains
- **Replace if-else chains** with dict lookups or polymorphism
- **Avoid deeply nested dicts/lists** -- use dataclasses/Pydantic instead
- **Keep comprehensions simple** -- if complex, use a loop

```python
# Bad: Deep nesting
def process_item(item):
    if item is not None:
        if item.is_valid():
            if item.has_data():
                return item.process()

# Good: Early returns
def process_item(item):
    if item is None:
        return None
    if not item.is_valid():
        raise ValueError("Invalid")
    if not item.has_data():
        return None
    return item.process()
```

```python
# Bad: if-else chain
def get_discount(user_type):
    if user_type == "premium":
        return 0.2
    elif user_type == "gold":
        return 0.15
    else:
        return 0

# Good: Dict lookup
DISCOUNT_RATES = {"premium": 0.2, "gold": 0.15, "silver": 0.1}
def get_discount(user_type):
    return DISCOUNT_RATES.get(user_type, 0)
```

```python
# Bad: Too many parameters
def create_user(name, email, age, address, city, country):
    ...

# Good: Group into dataclass
@dataclass
class UserData:
    name: str
    email: str
    age: int
    address: str
    city: str
    country: str

def create_user(data: UserData):
    ...
```

### Code Smells to Avoid

```python
# Bad: Magic numbers
if user.age > 18:
    ...

# Good: Named constants
MINIMUM_AGE = 18

    ...
```

```python

def add_item(item, items=[]):
    items.append(item)
    return items

# Good: Use None and create new list
def add_item(item, items=None):

def create_address(street, city, country, postal_code):
    save_to_db(street, city, country, postal_code)

class Address:
    street: str
    city: str
    country: str
    postal_code: str

def create_address(address: Address):

```

```python
# Bad: Hard to read method chain
result = df.loc[...].groupby(...).agg(...).assign(...).set_index(...).sum()

# Good: Format vertically
result = (
    df
    .loc[...]
    .groupby(...)
    .agg(...)
    .assign(...)
    .set_index(...)
    .sum()
)
```

```python
# Bad: Excessive comments stating the obvious
# Read CSV file
df = pd.read_csv("data.csv")
# Loop through rows
for row in df.iterrows():
    # Process the row
    process(row)

# Good: Code is self-explanatory
df = pd.read_csv("data.csv")
for row in df.iterrows():
    process(row)

# Good: Comments explain WHY, not WHAT
# Using chunked reading to avoid memory issues with 100GB+ files
for chunk in pd.read_csv("data.csv", chunksize=10000):
    process(chunk)
```

### Error Handling

**Principles:**
- **Fail fast**: Validate inputs early (in `__init__`, not methods), raise exceptions immediately
- **Use exceptions appropriately**: Raise for invalid input/unrecoverable errors; return `None`/`Optional` for expected failures
- **Be specific**: Use specific exception types, catch specific exceptions (not broad), create custom exceptions for domain errors
- **Clear messages**: Include context (IDs, values, parameters), suggest fixes when possible, never expose sensitive data
- **Handle correctly**: Let exceptions propagate if caller should handle, catch only what you can recover from, use `finally` for cleanup
- **Re-raise with context**: Use `raise ... from e`
- **Log with context**: Use appropriate levels (ERROR/WARNING/INFO), include debugging context (user_id, request_id)
- **User-facing errors**: Don't expose internals or stack traces, provide error codes and support IDs

```python
import logging
from pathlib import Path

logger = logging.getLogger(__name__)

# Custom exceptions
class DataValidationError(ValueError):
    """Raised when data validation fails."""

    pass

class ExternalServiceError(Exception):
    """Raised when external service call fails."""

    pass

# Fail fast -- validate early in __init__
class UserService:
    def __init__(self, config_path: str, api_client: APIClient):
        if not Path(config_path).exists():
            raise FileNotFoundError(f"Config not found: {config_path}")
        self.config = self._load_config(config_path)
        self.api_client = api_client

    def process_user(self, user_id: str) -> User | None:
        """Process user data with proper error handling.

        Returns None if user not found (expected case).
        Raises ExternalServiceError for API failures.
        """
        try:
            response = self.api_client.get(f"/users/{user_id}")

            if "email" not in response:
                raise DataValidationError(
                    f"Missing required field: email. "
                    f"Received fields: {list(response.keys())}"
                )

            logger.info("Successfully processed user %s", user_id)
            return User.parse(response)

        except NetworkError as e:
            logger.error(
                "Failed to fetch user from API",
                extra={
                    "user_id": user_id,
                    "endpoint": f"/users/{user_id}",
                    "request_id": response.headers.get("X-Request-ID"),
                },
                exc_info=True,
            )
            raise ExternalServiceError(f"Failed to fetch user {user_id}") from e

        finally:
            self.api_client.close_connection()

    def get_user_for_api(self, user_id: str) -> dict:
        """API endpoint handler -- returns user-facing errors."""
        try:
            user = self.process_user(user_id)
            if user is None:
                return {
                    "error": "User not found",
                    "message": f"No user exists with ID {user_id}",
                    "error_code": "USER_NOT_FOUND",
                }
            return {"user": user.dict()}

        except ExternalServiceError:
            request_id = generate_request_id()
            logger.error(
                "Service error for user %s",
                user_id,
                extra={"request_id": request_id},
            )
            return {
                "error": "Service temporarily unavailable",
                "message": "Please try again in a few minutes",
                "error_code": "SERVICE_UNAVAILABLE",
                "support_id": request_id,
            }
```

### Performance

**Principles:**
- **Measure first**: Profile before optimizing, identify actual bottlenecks not assumptions
- **When to optimize**: You have performance requirements (SLAs), you've measured the bottleneck, optimization doesn't significantly hurt readability
- **When "good enough" is enough**: Batch jobs overnight don't need sub-second optimization, readability often trumps micro-optimizations

```python
# Bad: O(n) linear search in list
if item in my_list:
    ...

# Good: O(1) membership test with set
if item in my_set:
    ...
```

```python
# Bad: Repeated calculations in loop
for item in items:
    max_value = calculate_max(data)  # Called every iteration
    if item > max_value:
        process(item)

# Good: Calculate once outside loop
max_value = calculate_max(data)
for item in items:
    if item > max_value:
        process(item)
```

```python
# Bad: Loads everything in memory
def get_all_records():
    return [process(record) for record in fetch_records()]

# Good: Yields one at a time (generator)
def get_all_records():
    for record in fetch_records():
        yield process(record)
```

```python
# Bad: N+1 query problem
users = User.query.all()
for user in users:
    user.orders  # Separate query for each user

# Good: Eager loading
users = User.query.options(joinedload(User.orders)).all()
```

## Docker and Containerization

### Dockerfile Best Practices

- **Order instructions**: least to most frequently changing (dependencies first, code last)
- **Use minimal base images**: `python:3.11-slim` not `ubuntu`
- **Combine RUN commands** to reduce layers
- **Use `.dockerignore`**: exclude `__pycache__`, `.git`, `.env`, `node_modules`
- **Use specific tags**: `python:3.11-slim` not `python:latest`
- **Don't run as root** -- create non-root user
- **Clean package caches** after install (`rm -rf /var/lib/apt/lists/*`)

```dockerfile
# GOOD: Layer caching -- dependencies first, code last
FROM python:3.11-slim
WORKDIR /app

COPY requirements.txt .
RUN pip install -r requirements.txt

COPY . .
CMD ["uv run", "python", "app.py"]
```

### Multi-Stage Builds

Use multi-stage builds for smaller, more secure images:
- **Build stage**: includes compilers, dev tools
- **Runtime stage**: only runtime dependencies
- Results in 80-90% smaller images

### Configuration and Secrets

- **Never hardcode secrets** in Dockerfiles or images
- Use **environment variables** for runtime config
- Use `.env` files for local dev (never commit)
- **Mount secrets at runtime**, don't bake into images

### Container Persistence

Containers are **ephemeral** -- data is lost when removed:
- Use **volumes** for persistent data (models, logs)
- Use **cloud storage** (Azure Blob) for production artifacts
- Use **model registries** (MLflow) for ML models
